# JS-Bridge Security Audit — 2026-06-22

Audit ticket: **AUDIT-L01** (docs-only, no code changes).

This document reviews the **5 JavaScript bridge registration sites** in
the ForPDA Android client. It does **not** modify any production code;
its purpose is to record the current exposure surface and to flag any
gaps that a future hardening pass should close.

## 1. Background

The app uses three distinct `WebViewSecurityProfile`s
(see `ExtendedWebView.kt`):

| Profile | JS enabled | `IBase` bridge allowed? | When used |
| --- | --- | --- | --- |
| `TRUSTED_LOCAL_TEMPLATE` | yes | **yes** | ForPDA forum theme (locally templated HTML, `4pda.to/forum/…`) |
| `TRUSTED_STATIC_ARTICLE` | yes | **no** | News articles, search results, forum rules (server-rendered HTML, but trusted origin) |
| (any other) | n/a | n/a | External URLs — bridge is **never** attached. |

The guards in `ExtendedWebView.enableBaseBridge()` enforce this:

```249:250:app/src/main/java/forpdateam/ru/forpda/ui/views/ExtendedWebView.kt
        addJavascriptInterface(this, IBase.JS_BASE_INTERFACE)
```

`enableBaseBridge()` is a no-op unless
`securityProfile == TRUSTED_LOCAL_TEMPLATE`. The specialized
interfaces (e.g. `ThemeJsInterface`, `SearchJsInterface`,
`QmsChatJsInterface`, `INews`, `IForumRules`) are registered by the
fragments directly, but only after the WebView's `init(profile)` call
has been made with the appropriate profile.

## 2. The 5 bridge registration sites

### 2.1 `ExtendedWebView.kt:249` — base `IBase` bridge

- **Interface**: `IBase`
- **JS name**: `IBase`
- **Methods**: `playClickEffect()`, `domContentLoaded()`, `onPageLoaded()`
- **Exposure**: Granted by `enableBaseBridge()`. The method is a no-op
  unless `securityProfile == TRUSTED_LOCAL_TEMPLATE`.
- **Risk**: **Low**. Methods are side-effect free aside from
  `playClickEffect()` (sound/haptic) and lifecycle flags. None of them
  take string parameters, so a `javascript:`-URL injection cannot
  reach them with attacker-controlled data.
- **Recommendation**: ✅ keep as-is. The hard `securityProfile` guard is
  the correct model.

### 2.2 `ThemeBridgeHandler.kt:34-35` — theme-specific bridge

- **Interfaces**: `ThemeViewBridge` (compatibility stub),
  `ThemeJsInterface` (`IThemePresenter`).
- **JS names**: `IThemeView`, `IThemePresenter`.
- **Exposure**: Granted by `ThemeBridgeHandler.init()` after the host
  fragment has called `webView.init(WebViewSecurityProfile.TRUSTED_LOCAL_TEMPLATE)`.
- **Risk surface**:
  - `ThemeViewBridge.callbackUpdateHistoryHtml(value: String)` is a
    **compatibility no-op** (line 75-78). It accepts a string and
    discards it. This is fine but slightly misleading — a future reader
    may assume it is wired somewhere. **Document or remove**.
  - `ThemeJsInterface` exposes ~30 methods (post edit, scroll, reply,
    highlight, etc.) that are callable from JS in the theme. Several
    take `String` parameters that are eventually rendered as HTML or
    passed to coroutines. The string values originate from
    JS that **the client itself generated** (the theme template is a
    local asset, not a remote page), so a JS-injection attack would
    require either a malicious theme asset (delivered via app update,
    not via the network) or a logic bug in the local template.
- **Risk**: **Low–Medium**. The local template trust model is sound,
  but the no-op `callbackUpdateHistoryHtml` should be removed or
  annotated so that it isn't mistaken for a real sink.
- **Cleanup contract**: `ThemeBridgeHandler.cleanup()` removes both
  interfaces and calls `removeBaseBridge()`. This is the **only** place
  in the codebase that does a complete 3-way bridge teardown.

### 2.3 `SearchFragment.kt:351` — `SearchJsInterface`

- **Interface**: `SearchJsInterface`
- **JS name**: `SEARCH_JS_INTERFACE` (private constant).
- **Profile**: `TRUSTED_STATIC_ARTICLE` (server-rendered HTML, JS
  allowed, `IBase` forbidden).
- **Methods**: navigation helpers (open result, prev/next page).
- **Risk**: **Low**. Search results originate from the 4PDA search API
  and are rendered into HTML client-side. The `javascript:` content
  inside the rendered HTML is generated by the client, not by the
  remote server. Any string parameters are URL fragments that pass
  through `Uri.parse` validation before being navigated to.
- **Recommendation**: ✅ keep. Add a code comment that ties the
  registration to `WebViewSecurityProfile.TRUSTED_STATIC_ARTICLE`
  (already done in the file-level KDoc on the fragment).

### 2.4 `ArticleContentFragment.kt:2999` — `INews` bridge

- **Interface**: `this` (the fragment).
- **JS name**: `INews`.
- **Profile**: `TRUSTED_STATIC_ARTICLE`.
- **Methods**: ~30 (comment likes, replies, open image/video, send
  poll, open taxonomy, etc.). Most are wrapped in `webView.runInUiThread`
  and most take a string `id` that is then matched against an
  in-memory comment map. Methods that take URLs (e.g. `openImage`,
  `openExternalBrowser`, `openArticleLink`, `openVideo`, `openTaxonomy`)
  route through the existing `SystemLinkHandler` / `LinkHandler` chain
  which is already validated.
- **Risk**: **Low**. The "missing context" guards (`if (context == null)
  return`) are present on every method that touches the fragment
  lifecycle. The article content is locally generated HTML, not
  remote.
- **Recommendation**: ✅ keep. Watch for new `@JavascriptInterface`
  methods that forget the `context == null` guard — this is the
  most likely future regression point.

### 2.5 `QmsChatFragment.kt:225,2472` — `QmsChatJsInterface`

- **Interface**: `QmsChatJsInterface`
- **JS name**: `JS_INTERFACE` (private constant on the fragment).
- **Profile**: **not explicitly set** by the fragment — the chat
  WebView is constructed with `ExtendedWebView(requireContext())` and
  **does not call `webView.init(profile)`**. This means
  `securityProfile` defaults to whatever the `ExtendedWebView`
  default is, which should be checked.
- **Risk**: **Medium–High**. The QMS chat HTML is rendered from
  server content (4PDA QMS API), and the bridge is registered on the
  same WebView. If the QMS content is ever served from an untrusted
  origin, the bridge methods (which include send-message and read-state
  toggles) become attackable. Even if the QMS content is currently
  always local-template, the lack of an explicit `init(profile)` call
  is a smell.
- **Recommendation**: 🛑 **add an explicit `webView.init(profile)` call
  to `QmsChatFragment.onViewCreated`** to lock in the trust model. The
  expected profile is `TRUSTED_LOCAL_TEMPLATE` (chat HTML is
  server-rendered into a local template), or a new
  `TRUSTED_QMS_CHAT` profile if the chat HTML origin needs its own
  tier. This is the **only material finding** in this audit.
- **Renderer recovery**: lines 2459-2476 correctly cancel the old
  `jsInterface` before re-attaching it to the new WebView. This
  prevents stale callbacks from reaching a cancelled presenter.

### 2.6 (additional) `ForumRulesFragment.kt:102` — `IForumRules` bridge

- **Interface**: `this` (the fragment).
- **JS name**: `JS_INTERFACE` (private constant).
- **Profile**: `TRUSTED_STATIC_ARTICLE` (explicitly set on line 86).
- **Methods**: ~5 (`copyRule`, `openLink`, etc.). The audit ticket
  listed this site as the 5th; the body of the bridge is the smallest
  of the five.
- **Risk**: **Low**. Forum rules content is local-template HTML.

## 3. Cross-cutting concerns

1. **Cleanup symmetry.** Every fragment that registers a bridge
   should remove it on `onDestroyView`. The `ThemeBridgeHandler`
   shows the right pattern (3-way cleanup with `removeBaseBridge()`).
   `ArticleContentFragment`, `QmsChatFragment`, `SearchFragment`,
   and `ForumRulesFragment` rely on WebView destruction to drop the
   bridge implicitly (`removeJavascriptInterface` happens when the
   WebView is GC'd). This is acceptable for these fragments because
   none of them rebind the WebView after `onDestroyView`.

2. **`context == null` guards.** Present on every `@JavascriptInterface`
   method that touches the fragment's context. This is the
   audit's primary defense against late callbacks after `onDestroy`.

3. **No string-to-code evaluation.** None of the five bridges accept a
   `javascript:` URL or a `String` parameter that is later passed to
   `WebView.loadUrl(...)` or `evaluateJavascript(...)` without
   validation. The local-template and trusted-article content models
   keep the trust boundary narrow.

4. **Origin validation.** The app does **not** enforce
   `WebView.setWebContentsDebuggingEnabled(false)` in release builds
   for any of the five sites. The production build relies on
   `BuildConfig.DEBUG` elsewhere; verify this is the case for the
   WebView settings. (Out of scope for this docs-only audit but worth
   a future check.)

## 4. Findings & recommendations

| Severity | Site | Recommendation |
| --- | --- | --- |
| 🛑 Medium–High | `QmsChatFragment.kt:225,2472` | Add explicit `webView.init(WebViewSecurityProfile.TRUSTED_LOCAL_TEMPLATE)` (or a new `TRUSTED_QMS_CHAT` profile). |
| 🟡 Low | `ThemeBridgeHandler.kt:75-78` | Remove the no-op `callbackUpdateHistoryHtml` or add a clear `// intentionally empty for legacy template compatibility` comment. |
| ✅ Low | `ExtendedWebView.kt:249` (`IBase`) | Keep; the `TRUSTED_LOCAL_TEMPLATE` guard is correct. |
| ✅ Low | `SearchFragment.kt:351` | Keep; `TRUSTED_STATIC_ARTICLE` is correct. |
| ✅ Low | `ArticleContentFragment.kt:2999` | Keep; `TRUSTED_STATIC_ARTICLE` is correct. |
| ✅ Low | `ForumRulesFragment.kt:102` | Keep; `TRUSTED_STATIC_ARTICLE` is correct. |
