# Проверка фоновых уведомлений ProPDA

Дата проверки: 20 июля 2026 года.

## Область проверки

Проверен полный путь фоновой доставки уведомлений:

- изменение настроек и планирование фоновых задач;
- `AlarmManager` и `WorkManager`;
- фоновый `EventsCheckWorker`;
- загрузка и расшифровка cookies;
- проверка QMS, избранных тем, упоминаний и изменений шапки;
- WebSocket и foreground-service;
- публикация системных уведомлений;
- диагностический журнал и системные ограничения Android/OEM.

Фокусные unit-тесты пакета уведомлений успешно прошли:

```text
./gradlew :app:testStableDebugUnitTest --tests 'forpdateam.ru.forpda.notifications.*'
BUILD SUCCESSFUL
```

При этом существующие тесты преимущественно проверяют форматирование уведомлений, ID,
группировку и отдельные условия запуска. Они не моделируют одновременный запуск двух
workers, непарсящийся HTTP 200, смерть процесса и частичный отказ сетевых источников.

## Текущая схема фоновой доставки

После ухода приложения в фон обычный WebSocket остаётся активным примерно 45 секунд.
Дальнейшая доставка зависит от одного из фоновых контуров:

1. точный или неточный `AlarmManager`;
2. одноразовый expedited `EventsCheckWorker`;
3. резервный периодический WorkManager с интервалом `2 × настройка`;
4. опциональный foreground-service в режиме «Постоянное соединение».

Без режима «Постоянное соединение» мгновенная доставка в длительном фоне архитектурой
не гарантируется. Нормальная задержка может составлять от 15–60 минут и больше под
ограничениями Doze или OEM.

## Найденные узкие места

### P1. Snapshot обновляется, даже если уведомление не показано

Файл: `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckWorker.kt`,
участки около строк 261–287 и 315–331.

Результат `NotificationPublisher.publish()` игнорируется, после чего новый серверный
snapshot или множество ключей упоминаний сохраняется безусловно.

`publish()` возвращает `null`, если:

- не выдано разрешение `POST_NOTIFICATIONS`;
- уведомления приложения выключены в Android;
- главный переключатель уведомлений выключен в момент публикации.

При следующем проходе событие уже считается обработанным. После выдачи разрешения или
включения уведомлений оно повторно показано не будет.

Рекомендация: разделить состояния `fetched`, `filtered`, `published` и `acknowledged`.
Продвигать delivery-состояние только после успешной публикации либо явно зафиксировать
политику отказа от накопленных событий.

### P1. Пустой ответ неотличим от ошибки парсинга

Файлы:

- `app/src/main/java/forpdateam/ru/forpda/model/data/remote/api/events/NotificationEventsApi.kt`;
- `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckWorker.kt`.

Inspector разбирается регулярными выражениями. Пустой список может означать как реальное
отсутствие событий, так и один из следующих случаев:

- Cloudflare или защитная страница;
- HTML формы входа;
- rate-limit;
- изменение формата ответа 4PDA;
- обрезанный ответ;
- отсутствие совпадений с регулярным выражением.

Если `AuthHolder` всё ещё считает пользователя авторизованным, воркер принимает пустой
список за корректный результат и сохраняет его как новый snapshot. Пока ответ не парсится,
уведомлений не будет. После восстановления ответа вся непрочитанная выдача может быть
ошибочно воспринята как новая.

Рекомендация: перед обновлением snapshot проверять HTTP status, content type, сигнатуру
inspector-ответа, отсутствие login/Cloudflare HTML и ожидаемую структуру данных.

### P1. Дедуп alarm и periodic worker не атомарен

Файл: `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckWorker.kt`,
участок около строк 108–125.

Периодическая и alarm-работа имеют разные unique-work имена:

- `events_check_periodic`;
- `events_check_alarm`.

Поэтому они могут выполняться одновременно. Дедуп реализован отдельным чтением и записью
`lastCheckAt` в `SharedPreferences`. Два воркера способны одновременно прочитать старое
значение, пройти проверку и начать сетевой цикл.

Последствия:

- двойные сетевые запросы;
- повторные публикации;
- конкурирующее обновление snapshot;
- нестабильная диагностика.

Рекомендация: использовать один общий unique-work контур либо атомарную блокировку/lease,
действующую для обоих источников запуска.

### P1. Успех одного источника скрывает сбой другого

Файл: `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckWorker.kt`,
участок около строк 131–165.

Retry назначается только тогда, когда не сработал ни один сетевой источник. Например:

1. QMS успешно вернул пустой список;
2. запрос упоминаний завершился `IOException`;
3. `anyNetworkSuccess` становится `true`;
4. воркер возвращает `Result.success()`.

Упоминания будут повторно проверены только через следующий полный интервал. Ошибки проверки
шапок `HAT` вообще не участвуют в общей логике retry.

Рекомендация: хранить результат, backoff и время последней успешной проверки отдельно для
каждого источника.

### P2. Проверяется только первая страница упоминаний

Файл: `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckWorker.kt`,
вызов `mentionsRepository.getMentions(0)`.

Если между проверками накопилось больше упоминаний, чем помещается на первой странице,
более старые события не будут обнаружены. На последующих проходах они могут оставаться на
второй странице и никогда не попасть в фоновую проверку.

Рекомендация: загружать страницы до первого известного ключа, пустой страницы или безопасного
ограничения числа страниц.

### P2. Первый успешный проход намеренно не уведомляет

Файл: `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckWorker.kt`,
участок около строк 215–234.

Для QMS, избранного и упоминаний первый фоновый проход только создаёт начальный snapshot.
Уже накопленные события не показываются. Это сознательная защита от массовой выдачи старых
уведомлений, но после установки, очистки данных или миграции пользователь может принять такое
поведение за поломку.

Рекомендация: явно объяснять это в диагностике или предлагать пользователю политику первого
запуска.

### P2. После перезагрузки точный alarm не восстанавливается напрямую

В приложении нет собственного `BOOT_COMPLETED` receiver для немедленного восстановления
alarm-цепочки. WorkManager должен восстановить сохранённую периодическую работу, а затем
воркер повторно поставит alarm. Резервный периодический интервал равен `2 × настройка`.

После перезагрузки первый сигнал может задержаться примерно на 30–120 минут плюс задержка
Doze/OEM.

Рекомендация: добавить boot receiver, который безопасно перепланирует alarm и periodic work
после загрузки устройства и обновления приложения.

### P2. Точный alarm не всегда является точным

Файл: `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckAlarm.kt`.

При отсутствии права на exact alarm код переходит на `setAndAllowWhileIdle()`. В этом случае:

- alarm может ждать maintenance window;
- OEM может дополнительно задержать или заблокировать запуск;
- expedited WorkManager может исчерпать квоту и стать обычной работой;
- после force-stop оба контура не работают до ручного запуска приложения.

Это преимущественно ограничение Android, но интервал в настройках не следует представлять
как гарантированное время доставки.

### P2. Авторизация зависит от асинхронной загрузки cookies

Файлы:

- `app/src/main/java/forpdateam/ru/forpda/client/CookieManager.kt`;
- `app/src/main/java/forpdateam/ru/forpda/notifications/EventsCheckWorker.kt`.

В холодном процессе воркер ждёт загрузку зашифрованных cookies до 10 секунд. Возможные
сценарии отказа:

- Keystore или `EncryptedSharedPreferences` не успевают;
- cookies невозможно расшифровать;
- `member_id` присутствует локально, но серверная сессия истекла;
- сервер возвращает гостевую страницу, а парсер принимает её за пустой результат.

Последний случай особенно опасен из-за возможного обновления snapshot пустыми данными.

### P2. Долгий фон по умолчанию работает через polling, а не WebSocket

Файлы:

- `app/src/main/java/forpdateam/ru/forpda/model/repository/events/EventsRepository.kt`;
- `app/src/main/java/forpdateam/ru/forpda/App.kt`;
- `app/src/main/java/forpdateam/ru/forpda/notifications/NotificationsService.kt`.

Обычный WebSocket после ухода приложения в фон имеет короткий grace period. Постоянный
WebSocket требует отдельной пользовательской настройки и foreground-service. Без неё
уведомления в длительном фоне приходят только через polling и неизбежно имеют задержку.

## Ограничения Android и OEM

Код уже предусматривает:

- запрос `POST_NOTIFICATIONS`;
- отображение состояния системных каналов;
- переход к исключению из оптимизации батареи;
- ссылки на OEM-настройки автозапуска;
- персистентный диагностический журнал;
- резервный WorkManager;
- ожидание загрузки auth-cookies;
- cache-buster для inspector-запросов.

Однако приложение не может программно обойти:

- принудительную остановку приложения;
- запрещённый OEM-автозапуск;
- режимы спящих приложений Samsung и аналогичные механизмы;
- отозванное право exact alarm;
- выключенные системные каналы уведомлений;
- агрессивную заморозку процессов прошивкой.

## Интерпретация диагностического журнала

Журнал доступен в приложении: «Настройки → Уведомления → Диагностика фоновых проверок».

| Запись или ситуация | Вероятная причина |
|---|---|
| Нет `alarm: fired` и `worker: run start` | Android/OEM не запускает фон |
| Есть `alarm: fired`, но нет `worker: run start` | WorkManager не запустил queued work или работа блокируется |
| `cookie hydration timeout` | Keystore/cookies не загрузились вовремя |
| `skip (not authorized)` | Приложение считает сессию неавторизованной |
| `QMS/THEME total=0` при наличии непрочитанного | Гостевой, защитный или непарсящийся ответ |
| `publish blocked: no POST_NOTIFICATIONS` | Публикация запрещена; событие сейчас может быть потеряно окончательно |
| `channel ... disabled` | Категория отключена в системных настройках Android |
| Частые `skip (checked Ns ago)` | Alarm и periodic запускаются рядом по времени |
| Ошибка только одного источника без retry | Частичный сетевой сбой скрыт успехом другого источника |

## Рекомендуемый порядок исправлений

1. Не продвигать delivery-состояние при неуспешной публикации.
2. Валидировать inspector-ответ перед сохранением snapshot.
3. Добавить общий атомарный lock для alarm и periodic worker.
4. Разделить retry и отметки успеха по источникам.
5. Добавить пагинацию фоновой проверки упоминаний.
6. Восстанавливать alarm после загрузки устройства.
7. Расширить диагностику: HTTP status, тип и размер ответа, число совпадений парсера и
   фактический результат публикации.

## Итог

Наиболее вероятны два внешне одинаковых режима отказа:

1. воркер вообще не запускается из-за Doze/OEM/force-stop;
2. воркер запускается, но получает или разбирает ответ как пустой и поэтому не публикует
   уведомления.

Самая опасная внутренняя ошибка — обновление snapshot независимо от фактического результата
публикации. Она превращает временный запрет уведомлений или системный отказ в окончательную
потерю события.
