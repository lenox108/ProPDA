# Отчёт по новым логам уведомлений ProPDA

Дата анализа: 20 июля 2026 года  
Назначение: отдельное дополнение к основному аудиту для передачи ИИ-агенту, который
дорабатывает уведомления.

## 1. Новые логи

Ключевые участки:

```text
01:20:13 alarm: fired (enabled=true)
01:20:13 worker: run start
01:20:13 QMS: total=0 new=0 published=0
01:20:13 THEME: total=43 new=2 published=2

01:22:49 scheduler: cancelled (main=false bg=true)

11:50:03 boot: MY_PACKAGE_REPLACED enabled=false
13:03:46 boot: MY_PACKAGE_REPLACED enabled=false

13:03:56 scheduler: alarm every 15 min + safety-net 30 min
13:03:56 worker: run start
13:03:56 QMS: seeded 0 without notifying
13:03:56 THEME: seeded 41 without notifying

13:07:26 ws: cooldown 10min (round=1)
13:08:39 ws: disconnected code=null SocketTimeoutException
...
13:22:26 ws: cooldown 10min (round=1)

13:30:12 alarm: fired (enabled=true)
13:30:12 worker: run start
13:30:13 QMS: total=0 new=0 published=0
13:30:13 THEME: total=42 new=6 published=5 muted=1 stacked
```

## 2. Главный вывод

Фоновый polling на этом устройстве работает. Это подтверждено двумя независимыми проходами:

- в `01:20` alarm запустил worker, который нашёл и опубликовал 2 события;
- в `13:30` alarm запустил worker, который нашёл 6 событий, отфильтровал одно и передал
  остальные 5 в групповое уведомление.

Следовательно, неработающий WebSocket не блокирует основную фоновую доставку. Текущая проблема
WebSocket — отдельная: он безрезультатно расходует сеть и батарею, а его circuit breaker не
выдерживает обещанный cooldown.

## 3. Что означает каждая часть логов

### 3.1. Polling успешно работает

```text
alarm: fired (enabled=true)
worker: run start
THEME: total=42 new=6 published=5 muted=1 stacked
```

Цепочка прошла полностью:

```text
AlarmManager -> BroadcastReceiver -> expedited WorkManager -> inspector -> сравнение snapshot
-> фильтрация -> NotificationManager
```

`published=5 stacked` означает пять событий, представленных одним групповым/summary
уведомлением, а не обязательно пять отдельных карточек в системной шторке.

`total=42`, но `new=6` не является противоречием. `new` вычисляется не как простая разница
размеров списков. Событие считается новым по `sourceId`, timestamp и росту `msgCount`; заодно
другие элементы могли исчезнуть после прочтения.

### 3.2. Планировщик был выключен настройкой приложения

```text
scheduler: cancelled (main=false bg=true)
```

Это означает:

- главный внутренний переключатель уведомлений `notifications.main.enabled=false`;
- фоновая проверка отдельно оставалась включённой: `bg=true`;
- приложение штатно отменило periodic work и alarm.

Это не означает, что Android сам запретил уведомления. Значение `main=false` читается из
preferences приложения. В просмотренном коде не найден автоматический setter, который сам
переводит этот ключ в `false`; обычный источник изменения — переключатель в настройках.

Вероятная последовательность: старая версия выдала большую пачку уведомлений (`published=25`),
после чего переключатель был выключен. Это правдоподобная гипотеза, но лог сам по себе не
доказывает, кто именно изменил настройку.

### 3.3. Обновление приложения сохранило выключенное состояние

```text
boot: MY_PACKAGE_REPLACED enabled=false
```

`MY_PACKAGE_REPLACED` означает обновление APK поверх установленного приложения, а не reboot.
Preferences при package replace сохраняются, поэтому receiver увидел прежнее `main=false` и
правильно не восстановил расписание.

В логах видны как минимум два обновления APK: в `11:50` и `13:03`. Поэтому записи до и после
них могут относиться к разным версиям реализации уведомлений.

### 3.4. После повторного включения произошёл seed

```text
QMS: seeded 0 without notifying
THEME: seeded 41 without notifying
```

Worker не потерял авторизацию и получил корректные данные. Он намеренно создал начальную базу
сравнения, чтобы не выдать все 41 накопленное событие как новые.

Поскольку это произошло сразу после обновлений APK, вероятен миграционный эффект: в старой
установке snapshot уже существовал, а новый boolean seed-флаг появился со значением `false`.
Первый запуск новой версии снова выполнил seed. Это защищает от burst, но может скрыть события,
которые действительно появились между последней проверкой старой версии и первым seed новой.

Требуется миграция: если старый snapshot уже существует, новый seed-флаг должен считаться
инициализированным.

## 4. Новая подтверждённая проблема: WebSocket cooldown сбрасывается

Лог обещает паузу 10 минут:

```text
13:07:26 ws: cooldown 10min (round=1)
```

Но следующая неудачная попытка зафиксирована через 73 секунды:

```text
13:08:39 ws: disconnected code=null SocketTimeoutException
```

Далее попытки продолжаются, и в `13:22:26` снова появляется:

```text
ws: cooldown 10min (round=1)
```

То есть circuit breaker фактически не обеспечивает десятиминутную тишину, а номер раунда
сбрасывается вместо роста `10 -> 20 -> 40 -> 60`.

### 4.1. Причина в коде

`EventsRepository.onDisconnected()` правильно устанавливает:

```kotlin
wsCooldownRounds++
wsReconnectSuppressedUntil = elapsedRealtime() + cooldownMs
```

Но `EventsRepository.onAppForegrounded()` безусловно выполняет:

```kotlin
consecutiveWsFailures = 0
wsReconnectSuppressedUntil = 0L
wsCooldownRounds = 0
```

Поэтому обычный переход приложения в foreground полностью снимает защиту, даже если сеть не
менялась. После следующих четырёх ошибок создаётся новый `round=1`.

Есть и второй путь потери состояния: `wsReconnectSuppressedUntil` и `wsCooldownRounds` не
персистятся. При смерти и холодном перезапуске процесса cooldown тоже исчезает. Персистится
только `wsAttemptsSinceSuccess`, используемый для UI-статуса.

### 4.2. Дополнительный риск: накапливающиеся delayed reconnect jobs

Код создаёт отдельные coroutine jobs для:

- reconnect backoff;
- пробуждения в конце cooldown;
- форсированных стартов при foreground, смене сети и авторизации.

Backoff jobs не хранятся в одном поле и не отменяются при изменении состояния. Старые jobs
позже просыпаются и снова вызывают `start()`. Пока cooldown активен, общий gate их остановит,
но после преждевременного сброса cooldown несколько отложенных jobs могут продолжить попытки.

Это хорошо согласуется с нерегулярными интервалами новых таймаутов в логе.

### 4.3. Требуемое исправление

1. Не сбрасывать `wsReconnectSuppressedUntil` и `wsCooldownRounds` при каждом foreground.
2. Полностью сбрасывать circuit breaker только после реального `onConnected()`.
3. При подтверждённой смене сети разрешать одну probe-попытку, но не стирать историю ошибок.
4. Персистить wall-clock `cooldownUntil`, `cooldownRound` и время последнего успешного
   подключения. `elapsedRealtime` нельзя напрямую сохранять между reboot.
5. Хранить ровно один `reconnectJob`; перед созданием нового отменять предыдущий.
6. После каждого `delay` повторно проверять generation/token, foreground, сеть, авторизацию,
   активный socket и текущий cooldown.
7. При стабильно недоступном endpoint использовать более длинную шкалу, например
   `10 min -> 30 min -> 2 h -> 24 h`.
8. Если пользователь включил постоянный WebSocket, но endpoint в длительном cooldown, не держать
   foreground-service только ради безрезультатных соединений; оставить polling.

### 4.4. Критерии приёмки circuit breaker

- После `cooldown 10min` в течение десяти минут нет ни одного нового socket connect, если сеть
  действительно не сменилась.
- Background/foreground переход не обнуляет раунд.
- Перезапуск процесса не обнуляет действующий cooldown.
- Второй раунд журналируется как `round=2` и имеет увеличенную длительность.
- Одновременно существует не более одной запланированной reconnect job.
- Успешный `ws: connected` полностью сбрасывает failure count и cooldown.
- Явная ручная кнопка «Проверить канал» может сделать одну диагностическую попытку, не запуская
  бесконечный reconnect loop.

## 5. Новая проблема диагностики alarm: видна значительная задержка

Расписание включилось в:

```text
13:03:56 scheduler: alarm every 15 min + safety-net 30 min
```

Ближайший alarm ожидался примерно в `13:18:56`, но сработал в:

```text
13:30:12 alarm: fired (enabled=true)
```

Отклонение от ожидаемого времени — примерно 11 минут 16 секунд. Полный интервал от записи
планировщика до фактического alarm — 26 минут 16 секунд.

Это не доказывает ошибку приложения. Возможные причины:

- `canScheduleExactAlarms()` вернул `false`, использован `setAndAllowWhileIdle()`;
- Doze/OEM задержал неточный alarm до maintenance window;
- другой внутренний вызов `schedule()` с тем же `PendingIntent` позже перенёс время;
- системное ограничение exact alarms.

Текущий пользовательский журнал пишет только «alarm every 15 min» и не сообщает:

- `exact=true/false`;
- рассчитанный `triggerAt`;
- кто вызвал повторное планирование;
- насколько поздно фактически пришёл alarm.

### Требуемое улучшение логирования alarm

При постановке:

```text
alarm: armed source=settings interval=15 exact=true triggerAt=13:18:56
```

При срабатывании:

```text
alarm: fired enabled=true expectedAt=13:18:56 lateBy=676s
```

`expectedAt` следует сохранять в preferences вместе с generation расписания. Это позволит
отличать задержку Android от повторного переноса самим приложением.

## 6. Разница поведения старой и новой версии

До package replace:

```text
THEME: total=42 new=28 published=25 muted=3
```

В строке нет `stacked`, хотя событий больше порога. Это похоже на старую версию до группировки
или на версию с другим форматом диагностики.

После package replace:

```text
THEME: total=42 new=6 published=5 muted=1 stacked
```

Новая защита от notification burst уже работает. Поэтому нельзя объединять весь журнал в один
эксперимент без учёта версии APK. В диагностику необходимо добавить:

```text
app: versionName=... versionCode=... commit=...
```

хотя бы один раз в начале экспортируемого журнала и после `MY_PACKAGE_REPLACED`.

## 7. Приоритет новых задач для ИИ-агента

### P1

1. Исправить сброс WebSocket circuit breaker в `onAppForegrounded()`.
2. Персистить cooldown и номер раунда между процессами.
3. Сделать единственную отменяемую reconnect job вместо множества независимых delayed jobs.

### P2

4. Добавить точную диагностику постановки и опоздания alarm.
5. Добавить безопасную миграцию seed-флагов при существующих старых snapshots.
6. Добавить versionName/versionCode/commit в экспорт диагностического журнала.

### P3

7. Увеличивать cooldown до часов/суток, если `app.4pda.to` стабильно недоступен.
8. При недоступном realtime явно показывать: polling работает, ожидаемая задержка — N минут плюс
   возможная задержка Android/Doze.

## 8. Итоговая оценка после новых логов

Положительное:

- авторизация в новых проходах доступна;
- alarm способен разбудить приложение;
- worker получает QMS/THEME;
- сравнение новых тем работает;
- mute-фильтр работает;
- новая stacked-публикация работает;
- WebSocket не является единственной точкой отказа.

Проблемы:

- мгновенный канал не установил ни одного соединения;
- circuit breaker сбрасывается и не останавливает reconnect loop;
- alarm опоздал примерно на 11 минут относительно заявленного интервала;
- seed после обновления мог подавить действительно новые события;
- диагностике не хватает версии APK и фактических параметров alarm.

Главный практический вывод: не нужно переписывать polling из-за этих логов — он доказанно
работает. Сначала следует остановить WebSocket reconnect storm, сделать cooldown устойчивым и
добавить измерение реальной задержки alarm. После этого уже отдельно решать, жив ли legacy
endpoint `app.4pda.to` и стоит ли вообще сохранять постоянный мгновенный канал.
